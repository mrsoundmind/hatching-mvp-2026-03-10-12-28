Wire the sliding-window failure counter (created in 35-01) into the actual provider-resolver fallback logic. When the chain exhausts 3 times within 60s, broadcast PROVIDER_DEGRADED to every connected WebSocket. When the next call succeeds, broadcast PROVIDER_RECOVERED. Additionally:
- Replay the current degraded state to any client that connects mid-outage so a page refresh during degradation still shows the banner.
- Wire the recovery-hook from providerHealthState.ts (registerRecoveryHook) so 35-05's /api/dev/force-recovery endpoint can deterministically broadcast PROVIDER_RECOVERED via forceRecoveryBroadcast() — keeps wave boundaries clean (35-05 only adds DEV-only HTTP endpoints; 35-02 owns ALL broadcaster wiring).

Purpose: This closes LLMUX-01 — typed PROVIDER_DEGRADED WS event when all providers fail, no raw HTTP 500 leakage, AND the late-join case where a user refreshes mid-outage. The resolver already tracks `attempted: ProviderId[]` per call; we observe that array's terminal state to drive the counter. Per D-13, a 429 that successfully falls back to the next provider is a "this single call succeeded" event, not a failure — only the throw at the end of the for-loop (line 263 / 299) counts as a chain exhaustion.

Output: Counter hooks added to generateChatWithRuntimeFallback() and streamChatWithRuntimeFallback() in providerResolver.ts; a new broadcastToAllSockets helper exposed from chat.ts; a global broadcaster registry in routes.ts that ALSO wires registerRecoveryHook for deterministic dev-side recovery; a late-join PROVIDER_DEGRADED replay in the wss.on('connection') handler; an integration test script proving emit semantics for 4 scenarios (degrade, recover-via-resolver, 429-not-counted, recover-via-forceRecoveryBroadcast).
