<objective>
Add the client-side toast banner for LLMUX-02 (banner appears) and LLMUX-03 (auto-dismiss within 5s of recovery signal). Two new WS message handlers in CenterPanel.tsx — one for `provider_degraded`, one for `provider_recovered` — backed by the existing shadcn toast infrastructure, plus a defensive dismiss-on-streaming_completed fallback per D-08.

Purpose: Per D-05 the client reuses `useToast` rather than building a new Banner component. Per D-06 toasts are persistent (`duration: Infinity`) until programmatically dismissed. Per D-08 dismissal can be signaled by `PROVIDER_RECOVERED` WS event from server (which 35-02 emits exactly once on next success) OR by a successful `streaming_completed` event — whichever arrives first. The 5-second LLMUX-03 criterion is measured from message arrival to dismissal — since dismissal happens synchronously in the WS handler, the latency is bounded by WS delivery (~ms) not a UI timer.

Critical eviction-aware design (the reason this plan is the way it is): `client/src/hooks/use-toast.ts` has `TOAST_LIMIT = 1` (line 8) and the reducer slices `[action.toast, ...state.toasts].slice(0, TOAST_LIMIT)` (line 79). Other toasts in CenterPanel.tsx fire on real events: "Task created" (line 829), "Team assembled" (line 839), "Task complete" (line 878), "Project memory updated" (line 991), "Slow down" (line 1009), "Task failed" (line 988). Any of these fired AFTER our degradation toast will EVICT the degradation toast from the visible list while leaving `degradationToastIdRef.current` holding a stale id. Naive gating on the ref would then (a) make the next PROVIDER_DEGRADED a no-op so the banner never re-shows, and (b) make PROVIDER_RECOVERED's `dismiss(stale_id)` a no-op silently. Solution below: NEVER gate the toast() call on the ref; let useToast dedupe via its TOAST_LIMIT; on recovery, dismiss-by-id (idempotent, safe-on-stale-id) and unconditionally clear the ref.

Output: Three additions to CenterPanel.tsx:
1. Extend the existing `const { toast } = useToast();` destructure at line 61 to `const { toast, dismiss } = useToast();` — single source of truth, no aliasing, no double-destructure.
2. `const degradationToastIdRef = useRef<string | null>(null)` near other refs.
3. Two new `else if (message.type === ...)` branches in the WS message dispatcher (line 1037+ after synthesis_completed), AND a small idempotent dismiss-block inside the existing `streaming_completed` handler.
</objective>
